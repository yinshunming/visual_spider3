## Context

构建一个基于 Java 的可视化爬虫 MVP，支持列表页和详情页的分离规则配置。系统需要提供可视化的选区功能，用户通过点击页面元素生成爬取规则，无需手动编写 XPath 或 CSS Selector。

**技术栈：**
- Java 21 + Spring Boot 3.x
- Thymeleaf（服务端渲染）
- MyBatis + PostgreSQL
- Quartz Scheduler
- Playwright for Java + Jsoup

**约束：**
- 第一版优先采用服务端渲染后台页面
- 不使用 iframe 嵌入第三方网站做选区
- 规则配置必须支持版本化（未来）
- 每个里程碑结束时必须能编译通过

## Goals / Non-Goals

**Goals:**
- 提供可视化的列表规则配置（容器选择器、分页选择器、列表项选择器）
- 提供可视化的详情规则配置（多字段选择器）
- 支持列表数据提取（标题、链接、时间）
- 支持详情数据提取（正文、作者、标签）
- 支持分页处理
- 支持 Quartz 定时调度
- 支持 Upsert 入库（幂等）
- 支持执行日志和手动重试

**Non-Goals:**
- 规则版本化管理（MVP 阶段不实现）
- 实时告警通知（MVP 阶段不实现）
- 分布式爬虫
- 代理池管理
- 反爬对抗策略
- AI 辅助规则推荐

## Decisions

### 1. 分层架构（标准三层）

**决定：** Web层 / Service层 / Repository层 / 引擎层

**理由：** 项目规模适中，标准分层清晰简单，适合 MVP 快速交付。

**备选：** 六边形架构 - 抽象程度高但复杂度也高，MVP 阶段不必要。

### 2. 列表规则和详情规则分离

**决定：** 列表规则和详情规则分别配置，独立管理

**理由：** 列表页和详情页的结构差异大，分离更灵活。任务配置中通过 list_rule_id 和 detail_rule_id 关联。

### 3. 规则设计简化（MVP）

**决定：** 一个字段一个规则，不做版本化

**理由：** MVP 快速交付，减少初始复杂度。用户通过选择候选列表中的一个来确定最终规则。

### 4. 可视化选区方案：Browser-in-Box

**决定：** Playwright 启动独立浏览器进程，截图 + 坐标映射，用户在截图上点击

**理由：** 项目约束不允许使用 iframe，Browser-in-Box 满足要求且实现复杂度适中。

**备选：** 独立桌面应用 - 体验更好但需要单独开发。

### 5. 入库策略：Upsert

**决定：** 根据唯一标识判断 Insert 或 Update

**理由：** 保证幂等性，支持重复执行不产生重复数据。

### 6. 失败排查：简化版

**决定：** 仅记录日志到数据库，不实现实时告警

**理由：** MVP 阶段快速交付，告警功能后续迭代。

## Risks / Trade-offs

| 风险 | 描述 | 缓解措施 |
|------|------|----------|
| 网站结构变化 | 目标网站改版导致规则失效 | 保留规则版本化扩展点 |
| 反爬限制 | 目标网站限制爬取 | 未来可扩展 User-Agent、代理池 |
| 大规模数据 | 列表数据量大时内存压力 | 分页处理 + 批量入库 |
| 规则冲突 | 多个字段选择器选择同一元素 | 用户选择时明确标注 |

## Migration Plan

MVP 分 5 个里程碑交付：

1. **M1: 项目骨架** - Spring Boot 项目初始化、数据库表创建、基础 CRUD
2. **M2: 列表规则** - Playwright 集成、可视化选区、列表规则预览
3. **M3: 详情规则** - 详情页规则配置、字段映射
4. **M4: 调度执行** - Quartz 集成、定时任务、数据入库
5. **M5: 失败排查** - 执行日志、手动重试

每个里程碑结束时必须能编译通过。
